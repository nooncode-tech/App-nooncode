# project.context.core.md

## Project identity
- Project: `nooncode-app`
- Product name in UI: `NoonApp`
- Type: Next.js web application
- Current stage: hybrid MVP with real Supabase auth/session, multiple persisted business slices, and active Maxwell Lead Engine V1 consolidation

## Primary purpose
- Confirmed: the app is a role-based sales-to-delivery workspace that combines lead management, delivery/project tracking, personal earnings/rewards, reporting, and an embedded AI copilot.

## Confirmed module map
- Public/login surface: `/`
- Dashboard home: `/dashboard`
- Internal updates feed: `/dashboard/updates`
- Sales modules: `/dashboard/leads`, `/dashboard/pipeline`
- PM inbound review module: `/dashboard/pm-queue`
- Outbound Maxwell Lead Engine V1: `/dashboard/leads`, `POST /api/maxwell/lead-searches`, and lead detail speech/feedback surfaces
- Delivery modules: `/dashboard/projects`, `/dashboard/tasks`
- Finance/personal modules: `/dashboard/earnings`, `/dashboard/rewards`, `/dashboard/reports`
- Admin module: `/dashboard/settings`
- AI assistant surface: dashboard-mounted Maxwell chat backed by `/api/maxwell`

## Confirmed stack snapshot
- Framework: Next.js 16 App Router
- Language: TypeScript
- UI runtime: React 19
- Package manager: pnpm
- Styling: Tailwind CSS v4 plus local shadcn/Radix-style component stack
- Auth/runtime dependency now present: Supabase SSR + Supabase JS

## Confirmed architecture shape
- Root layout resolves initial auth state server-side in `app/layout.tsx`.
- `AuthProvider` supports two modes: `supabase` when env is enabled and configured, `mock` as fallback.
- Dashboard route protection now exists at both middleware and client shell layers.
- Business-domain data for leads, projects, tasks, rewards, users, and points still lives in `lib/data-context.tsx`.
- `lib/dashboard-selectors.ts` remains the main selector/view-model layer over client data.
- Noon Website and Noon App are separate products. The website owns the public inbound/client/payment experience; Noon App owns collaborator operations and receives website handoffs through server-to-server webhooks.
- Maxwell Lead Engine V1 is App/outbound/seller-only. It must not be treated as the website inbound Maxwell, and it must not modify website, inbound, payments, workspace, developer board, post-payment handoff, earnings rules, or sensitive permissions unless explicitly scoped.
- Durable product context for Maxwell Lead Engine V1 now lives in `docs/product/maxwell-lead-engine-v1.md`; source PDFs are versioned under `docs/product/source/`.

## Active risks
- Mock-only surfaces remaining: rewards/points domain and the global users directory still read from in-memory state in `lib/data-context.tsx:442-443`. All other business data (auth, leads, proposals, projects, tasks, wallet, earnings) persists in Supabase.
- Local context files can drift quickly unless updated after each real phase.
- Bridge wallet (0025): conversión 1 crédito = $1.00 es **permanente como invariante** (per ADR-009 firmado 2026-05-13). `wallet_accounts` queda como única source of truth para escrituras nuevas; `user_wallets` queda como cache legacy read-only excepto por el dual-write existente en `request_lead_prototype`. Retiro completo de `user_wallets` se difiere a antes de v3 Phase 8. Riesgo permanente: cualquier escritura directa a `user_wallets` fuera del RPC bridgeado rompe la invariante silenciosamente.
- Legacy `/client/[token]` en App-nooncode: ruta placeholder pre-v3 reclasificada como deuda técnica (per ADR-010 firmado 2026-05-13). No recibe features nuevas. Removal scheduled cuando NoonWeb ship `/portal/[projectId]` (v3 Phase 2-6) o como parte de un cleanup explícito.
- Outbound Stripe Checkout creation in App (`app/api/payments/checkout/route.ts` + the `Crear/copiar link de pago` UI in `components/lead-detail.tsx`) is permitted under the **ADR-010 amendment 2026-05-14**: the route requires an internal-role principal, rejects inbound flows, the seller delivers the URL to the client out-of-band, and payment completion is verified via the Stripe webhook with no client-side authentication into App. Not a violation, but a constrained exception — any future re-introduction of a client-authenticated path into this route, or any leak of inbound flows past the rejection guard, would re-open the principle and must escalate.
- Stripe live keys pendientes de subir a Vercel antes del primer deploy a producción.
- Distributed rate limiter is verified end-to-end on Production as of 2026-05-15. `lib/server/api/rate-limit.ts` uses `@upstash/ratelimit` sliding window against the Upstash Redis instance provisioned via the Vercel Marketplace; `resolveDistributedRedisCredentials` reads `UPSTASH_REDIS_REST_URL` + `UPSTASH_REDIS_REST_TOKEN` first and falls through to `KV_REST_API_URL` + `KV_REST_API_TOKEN` (the env vars Vercel injects under its KV product branding for the same Upstash backend) when the UPSTASH_* pair is unset or empty. Production smoke test 2026-05-15: 80 parallel GETs against `/api/client/resolve` (limit 60/min, IP-keyed) returned exactly 60 × 404 + 20 × 429, and the Upstash dashboard recorded 253 commands (126 writes + 127 reads — ~3 ops per `limit()` call from the `slidingWindow` Lua script), confirming the limiter routed through Redis and not the in-memory fallback. Fail-open policy still logs `rate_limit.upstash.fallback` via `logger.warn` so future Upstash outages do not deny all rate-limited traffic. Remaining risk surface: **(a)** Upstash free tier exhaustion (current usage 253 / 500k commands per month) would silently shift the limiter into fail-open mode without alerting beyond the warn log — operator must monitor the Upstash dashboard and upgrade to paid plan if monthly usage approaches ~70% of the free cap; **(b)** per-user-id keying not implemented — limiter still uses client IP from forwarded headers; **(c)** Vercel Deployment Protection is active on the production deployment URL, which means future ops smoke tests must use the Protection Bypass for Automation token (header `x-vercel-protection-bypass`) or a custom-domain alias that is not protected.

Runbook — how to execute

Environment

  • pnpm dev against the Supabase dev/preview project (AUTH_MODE=supabase implied by env).
  • Chrome (or Edge / Firefox) with DevTools open in responsive mode.
  • Logged in as a role that has access to /dashboard, /dashboard/leads, and at least one visible lead (any role except a permission-denied subset).
  • Optional: real-device verification (iPhone / Android) recommended for tap-target sanity but not gating per spec § Recommended Testing Methodology.

Viewports to exercise

  1. 375 × 667 (iPhone SE 2nd gen) — primary gate. Must pass all in-scope criteria.
  2. 768 × 1024 (iPad portrait) — breakpoint boundary check (useIsMobile returns false at exactly 768 → desktop sidebar mode kicks in).
  3. 1280 × 800 — desktop regression check.

Horizontal-scroll quick check (paste in DevTools console at each viewport)

// returns true if body horizontal-scrolls
document.documentElement.scrollWidth > window.innerWidth

Expected result: false at every in-scope viewport for every in-scope route.

Methodology compliance

Per specs/fase-3-g21-mobile-responsive-fix.md § Recommended Testing Methodology, the iteration uses integration-first with browser validation as the gate. The spec also called for "one focused component test that the new mobile-header renders SidebarTrigger and that the trigger composition is functional inside SidebarProvider."

Methodology adjustment finding (recorded as test debt)

Finding ID T-G21-M1
Severity LOW (acceptable for this iteration)
Type methodology / test infra debt
Description The project has no React component testing infrastructure (no jsdom / no React Testing Library / no Vitest). All 56 pre-existing test files are .ts and run under tsx --test (Node's built-in runner) against server / lib / infra code. Introducing the spec-proposed component test that exercises SidebarProvider + SidebarTrigger behavior would require adding the DOM-test infrastructure as a dependency of G21 — that is iteration-scope creep and was not approved.
Mitigation applied Followed the existing project pattern set by B18 (tests/app/error-pages.test.ts): smoke import of the component module asserting the named export is a function. This catches missing exports, broken imports, and TypeScript breaks. Behavioral validation is fully delegated to browser validation (this document).
Recommended next action Open a separate iteration "Frontend testing infra" if/when broader component coverage becomes a recurring need (likely concurrent with the optional frontend-redesign track per docs/runbooks/frontend-redesign-playbook.md). Until then, document the gap and rely on browser validation + smoke imports.

Test debt summary

  • T-G21-M1 — DOM-level component test for the new mobile-header chrome is not added; smoke import only. Browser validation is the behavioral gate. See methodology adjustment above.
  • No other test debt introduced.
